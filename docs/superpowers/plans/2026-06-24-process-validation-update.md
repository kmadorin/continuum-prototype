# Process-Validation Update Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Fold all 7 research corrections (`docs/prompts/RESEARCH_FINDINGS_process-validation.md`) into the portal prototype and the user story map so the deal model reads credibly to a secondaries professional.

**Architecture:** The portal is a static front-end. `portal/shared/state.js` is the deal engine (no DOM) with a node self-test (`portal/shared/selftest.js`); `portal/shared/app.js` is the whole UI. The story map is generated by `docs/specs/gen_story_map.py`. We change the stage machine (insert an LPAC pre-elections consent gate, rename `cleared`→`leadSelected`, add a terminal `declined`), make lead-selection an explicit advisor action, add a three-way election (`status-quo`), reframe the fairness opinion as a supporting doc, add decline-to-proceed, then mirror all of it in the UI and the story map.

**Tech Stack:** Vanilla JS (no framework), Node for the engine self-test, Python 3 for the story-map generator, Excalidraw schema v2.

**Test strategy:** Engine changes are covered by `node portal/shared/selftest.js` (extend it; it must stay green at every commit). UI changes have no automated test — each UI task's verification step runs the self-test (regression) plus a concrete manual check in the browser. Commit after every task.
